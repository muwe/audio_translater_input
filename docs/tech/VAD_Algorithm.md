# VAD 语音活动检测算法

## 用途

Audio Translate Input 在用户按住 Fn 键时录制音频。许多录音是意外的——用户短暂点按了 Fn，或在安静的房间里按住了它。将这些空录音发送到 ASR 服务器会浪费配额、增加延迟，并返回无用的结果。

VAD 充当客户端侧的门控：它实时分析录制的音频，在上传前丢弃不包含有意义语音的录音。这节省了服务器成本，减少了网络流量，并避免了令人困惑的"空转录"结果。

---

## VAD 运行位置

VAD 完全在**子窗口**（`app_demo/lib/floating_window.dart`）中运行，位于 `_FloatingBarState` 类内部。它在录音期间运行，并在用户释放 Fn 键后立即做出通过/拒绝决定。

子窗口没有外部网络访问。如果 VAD 通过，音频文件路径通过 IPC（`upload_audio`）发送到主窗口。如果 VAD 拒绝，音频文件在本地删除，UI 静默返回空闲状态——不会发生 AI 调用。

---

## 语音帧计数方式

录音期间，`record` 包提供每 80 毫秒采样一次的振幅流：

```dart
_audioRecorder.onAmplitudeChanged(const Duration(milliseconds: 80))
```

每个样本报告 `amp.current`，单位为分贝（dBFS），范围大约从 -50（静音）到 0（最大值）。

对于每个样本（帧）：

1. **总帧计数器**递增：`_vadTotalFrames++`
2. **语音帧检测**：如果 `amp.current > -38.0 dB`，该帧被计为语音帧：`_vadActiveFrames++`

-38 dB 阈值是有意设置得较严格的。典型的背景噪音（风扇、空调、键盘敲击）约在 -45 到 -40 dB。正常音量的人声在麦克风距离处高于 -35 dB。-38 dB 的截止值位于这两个范围之间，以最小化环境噪音的误报。

**代码参考**（`floating_window.dart`，`_startAmplitudeMonitor` 方法，第 358-369 行）：

```dart
_vadTotalFrames++;
if (amp.current > -38.0) _vadActiveFrames++;
```

## 拒绝决策

录音停止后，`_stopAndUpload` 方法计算三个指标并检查所有三个阈值：

```dart
final activeRatio = _vadTotalFrames > 0 ? (_vadActiveFrames / _vadTotalFrames) : 0.0;

if (activeRatio < 0.15 || fileSize < 3000 || recordingDurationMs < 800) {
  // 拒绝：删除文件，返回空闲状态
}
```

### 阈值 1：语音帧比例 >= 15%

`activeRatio` 是振幅超过 -38 dB 的 80ms 帧所占比例。比例低于 0.15（15%）意味着录音绝大部分是静音或背景噪音。

**为什么是 15%：** 以每帧 80ms 计算，5 秒录音产生约 62 帧。15% 意味着至少约 9 帧（约 720ms）的语音内容。这足以包含一个短单词或短语。较低的阈值（如 5%）会让包含单次咳嗽或键盘敲击的录音通过。较高的阈值（如 30%）会拒绝用户在句子间停顿的录音。

### 阈值 2：时长 >= 800ms

`recordingDurationMs` 是从录音开始到停止的挂钟时间。短于 800ms 的录音几乎都是意外的 Fn 点按。

**为什么是 800ms：** Fn 键防抖窗口为 150ms，录音设置（创建文件、启动编码器）增加约 50-100ms 的开销。一个有意的"按住并说话"手势至少需要 500ms 的 Fn 按住加上一些语音。800ms 提供了舒适的余量。

### 阈值 3：文件大小 >= 3000 字节

`fileSize` 是磁盘上编码后的 M4A（AAC-LC，32kbps）文件大小。小于 3KB 的文件表示录音极短或编码器故障产生了近乎空的容器。

**为什么是 3KB：** 在 32kbps 比特率下，1 秒音频产生约 4KB 的编码数据。3KB 阈值捕获实际编码音频短于约 0.75 秒的录音（考虑容器开销）。这作为时长检查的辅助保护，捕获计时器报告时长长于实际编码内容的边缘情况。

---

## 拒绝后的处理

当任一阈值未满足时：

1. 音频文件从磁盘删除（`audioFile.delete()`）。
2. UI 直接转为 `BubbleIdle` 状态。
3. 悬浮条通过胶囊退出动画隐藏。
4. 不向主窗口发送 IPC 消息——主窗口永远不会知道曾有录音尝试。
5. 不向用户显示错误消息。该行为设计为"什么都没发生"的感觉，与用户意外点按的意图相符。

**拒绝时的日志输出：**
```
[VAD] Size: 1200 bytes, Time: 350ms, active ratio: 0.03 (1/30)
[VAD] content deemed invalid (too short or no speech) -> silently discarded!
```

---

## 接受后的处理

当所有三个阈值均满足时：

1. UI 转为 `BubbleProcessing` 状态（显示加载动画）。
2. 子窗口通过 IPC 向主窗口发送 `upload_audio`，包含音频文件路径、模式（润色/翻译）和录音时长。
3. 主窗口的 `AudioUploadService` 接管：直接调用 ASR/LLM API、获得转录结果、注入剪贴板并粘贴。

---

## 调优建议

当前阈值偏保守——倾向于上传可疑录音而非丢弃有效语音。如果用户报告过多空转录或 API 配额消耗过快，可考虑：

- **提高语音比例**从 15% 到 20-25%，以更积极地过滤。
- **提高 dB 阈值**从 -38 到 -35，要求更响亮的语音（但可能拒绝轻声说话的用户）。
- **添加最小语音帧数**（如无论比例如何至少 5 个活跃帧），以捕获非常短的录音恰好具有高比例的边缘情况。

任何阈值更改都应在多种麦克风类型（MacBook 内置、外接 USB、AirPods）和环境噪音条件（安静房间、咖啡厅、办公室）下进行测试。
